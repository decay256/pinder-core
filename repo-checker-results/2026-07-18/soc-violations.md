# Separation of Concerns Audit: soc-violations

Audited `pinder-core` and `pinder-web` at their current checked-out state for DB/query logic in controllers, business logic in UI, and layer ownership violations. Inspected GameApi controllers/services/data repositories, pinder-core gameplay/data/runner modules, frontend API clients, hooks, pages, and high-blast-radius game components. No U1 findings were found. No pinder-core-only production finding rose above the pinder-web issues below; pinder-core's file/JSON loaders and repository classes generally own their own data concerns.

### Finding 1: SessionsController owns turn-preview gameplay rules
**File**: `pinder-web/src/Pinder.GameApi/Controllers/SessionsController.cs:201`
**Issue**: `BuildTurnStateAsync` is a controller helper, but it computes option mechanics and corruption risk instead of delegating to a service/core projection: `int levelBonus = Pinder.Core.Progression.LevelTable.GetBonus(session.Player.Level);`, `int dc = defStats.GetDefenceDC(opt.Stat);`, `int tellBonus = opt.HasTellBonus ? 4 : 0;`, `var pairedShadow = StatBlock.ShadowPairs[opt.Stat];`, `RiskPct = CalculateRiskPct(shadowVal)`, and `EffectLabel = GetShadowEffectLabel(pairedShadow)`. The same file also embeds the risk formula and shadow copy in `CalculateRiskPct` / `GetShadowEffectLabel` at lines 274-296.
**Impact**: The HTTP controller is now a gameplay projection layer. Changes to shadow pairing, risk percent math, tell/triple bonuses, or effect labels must be remembered in controller code as well as pinder-core; otherwise `GET /api/sessions/{id}/turn` can show stale option previews while the engine resolves turns with different rules.
**Urgency**: U2 - escalated from U3 because this is the production turn-start hot path and stale previews can mislead player choices even when server resolution remains correct.
**Fixer-Agent Action Plan**: Introduce a `TurnStateProjectionService` or `DialogueOptionPreviewBuilder` outside the controller, preferably backed by pinder-core helpers for risk/effect semantics. Move `CalculateRiskPct` and shadow effect-label ownership into a domain/projection service, inject that service into `SessionsController`, and keep the controller to route/auth/status mapping. Add tests that compare projected `DialogueOptionDto` fields against pinder-core stat/shadow fixtures and update existing `Issue957_DialogueOptionRiskTests` to exercise the new service rather than duplicating controller formulas.

### Finding 2: UI predicts trap-disarm eligibility from stat strings
**File**: `pinder-web/frontend/src/components/OptionCard.tsx:24`
**Issue**: The UI decides whether an option disarms an active trap by normalizing display/wire strings: `const s = option.stat.toLowerCase().replace(/[_\s-]+/g, ''); return s === 'selfawareness' || s === 'sa';`. `OptionSelectionWidget` then renders the badge with `showSaDisarmBadge={trapActive && isSelfAwarenessOption(opt)}` at `pinder-web/frontend/src/components/OptionSelectionWidget.tsx:122`. The actual engine rule lives in pinder-core resolution code: `if (chosenOption.Stat == StatType.SelfAwareness && state.Traps.HasActive) { ... state.Traps.Clear(); }` in `pinder-web/pinder-core/src/Pinder.Core/Conversation/RollResolutionStage.cs:82`.
**Impact**: The frontend owns a gameplay rule and a set of accepted stat spellings. If trap clearing becomes item-, trap-, clear-method-, or backend-config-dependent, the UI can advertise "Disarms trap" for options that no longer clear traps, or fail to advertise a new valid clearing option.
**Urgency**: U3 - topic default; the server remains authoritative, but the player-facing affordance can drift.
**Fixer-Agent Action Plan**: Add an explicit backend-projected field such as `disarms_active_trap` or `trap_clear_preview` to `DialogueOptionDto`, computed from the same domain rule used by turn resolution. Replace `isSelfAwarenessOption` badge gating with the DTO field and keep string normalization only in display-name helpers. Add a frontend contract test proving the badge follows the DTO, and a backend test proving the DTO preview matches `RollResolutionStage` trap-clear behavior.

### Finding 3: Admin narrative harness controller orchestrates harness domain setup directly
**File**: `pinder-web/src/Pinder.GameApi/Controllers/AdminNarrativeHarnessController.cs:190`
**Issue**: The controller constructs model options and transport, loads pinder-core harness data, builds the confession menu, creates the pursuer actor, instantiates the runner, and executes it inline: `var (_, transport, _, _) = _llmFactory.Create(modelSpec, adapterOptions);`, `var character = await HarnessCharacterLoader.LoadAsync(...)`, `var baseDef = await HarnessCharacterLoader.LoadBaseGameDefinitionAsync(...)`, `var menu = ConfessionMenu.Build(...)`, `IPursuerActor pursuer = PursuerActorFactory.Create(...)`, and `var runner = new HarnessRunner(...); var result = await runner.RunAsync();`.
**Impact**: The HTTP layer owns narrative-harness orchestration, filesystem-backed harness loading, model transport setup, and raw-session projection. That makes the admin endpoint harder to test in isolation, prevents reuse by a background worker/CLI path, and spreads harness lifecycle decisions across controller and pinder-core tool classes.
**Urgency**: U3 - topic default; this is admin-only, but it is a broad controller ownership violation.
**Fixer-Agent Action Plan**: Extract a `NarrativeHarnessRunService` that accepts a validated request/options DTO and returns a typed run result plus operation telemetry fields. Keep controller code to model binding, auth, operation status mapping, and response projection. Unit-test the service with fake `ILlmTransport`, fake character loader, and fake pursuer factory; keep controller tests focused on route/status/error mapping.

### Finding 4: NarrativeTestbed page bypasses the frontend API layer for admin harness calls
**File**: `pinder-web/frontend/src/pages/NarrativeTestbed.tsx:98`
**Issue**: The page mixes UI state with raw HTTP calls even though `frontend/src/api/adminClient.ts` already centralizes admin JSON fetch behavior with `credentials: init.credentials ?? 'same-origin'` and `ApiClientError` conversion. The page directly calls `fetch('/api/admin/prompts/reload', ...)`, `fetch('/api/admin/narrative-harness/runs?limit=50')`, `fetch(\`/api/admin/narrative-harness/runs/${id}\`)`, and `fetch('/api/admin/narrative-harness/run', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify(...) })`.
**Impact**: Admin auth, request-id propagation, error parsing, credentials policy, and endpoint paths are split between UI components and API clients. Future changes to admin error shape or auth behavior must be patched in multiple places, and this page can diverge from the rest of the admin frontend.
**Urgency**: U3 - topic default; mostly maintainability and consistency, but it affects several admin workflows.
**Fixer-Agent Action Plan**: Add `frontend/src/api/narrativeHarnessClient.ts` (or extend `adminClient.ts`) with `reloadAdminPrompts`, `listNarrativeHarnessRuns`, `getNarrativeHarnessRun`, and `runNarrativeHarness` functions using `adminJsonFetch`. Move request/response interfaces there, replace raw `fetch` calls in `NarrativeTestbed.tsx`, and update page tests to mock the client functions instead of global fetch.

### Finding 5: MediaController owns Eigencore asset-store protocol details
**File**: `pinder-web/src/Pinder.GameApi/Controllers/MediaController.cs:57`
**Issue**: `MediaController` constructs the upstream Eigencore client request directly: `var client = _httpClientFactory.CreateClient();`, sets `client.DefaultRequestHeaders.Authorization`, builds multipart metadata with `asset_kind = "photobooth/v1"`, copies the uploaded file into a byte array, and posts to `await client.PostAsync($"{baseUri}/assets", content, ct)`. The download path likewise builds `GET {baseUri}/assets/{assetId}` in the controller at lines 133-141.
**Impact**: The controller owns upstream protocol, metadata schema, auth-header policy, upload body construction, and public URL derivation. Any asset-store API change must be coordinated inside HTTP route code rather than a focused media/asset service, and unit tests have to exercise controller plumbing to validate upstream request behavior.
**Urgency**: U3 - topic default; production-facing but primarily a layer ownership and maintainability issue.
**Fixer-Agent Action Plan**: Extract an `IMediaAssetStore` / `EigencoreMediaAssetStore` service that accepts the uploaded stream and returns an asset id/content stream result. Register a named `HttpClient` for asset-store calls, move metadata/auth/request construction into the service, and leave `MediaController` to validate route/form inputs and translate service outcomes to HTTP responses. Add service tests with a fake `HttpMessageHandler` and controller tests that assert only route/status mapping.

Suppressed by exception: Unsafe Leakage of GameApi Raw Exception/HTML Content (resp.text) to public-facing HTTP responses - would be U1

Further U3 instances exist in: `pinder-web/frontend/src/components/TopNav.tsx:69` (inline `/api/version` fetch outside an API client), `pinder-web/frontend/src/components/CharacterSheetModal.tsx:254` (component calls `apiFetch` for level-up instead of a dedicated session/player API helper).
