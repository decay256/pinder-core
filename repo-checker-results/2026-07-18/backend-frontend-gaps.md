# Backend/Frontend Integration Gaps

Inspected the current full repositories for `pinder-core` and `pinder-web`, focusing on pinder-core gameplay DTOs, `Pinder.GameApi` controllers/models, FastAPI proxy routers, Caddy routing, and React API clients/pages/components. I did not inspect or rely on other topic reports. Approved exception-class error leakage patterns were not raised.

### Finding 1: Narrative Testbed Calls a GameApi Reload Route That FastAPI Never Exposes
**File**: `pinder-web/frontend/src/pages/NarrativeTestbed.tsx:98`
**Issue**: The frontend save path performs a second reload call with `await fetch('/api/admin/prompts/reload', { method: 'POST', credentials: 'same-origin' })` after `putAdminContent('prompts-narrative', ...)`. The upstream GameApi endpoint exists as `[Route("api/admin/prompts")]` plus `[HttpPost("reload")]` in `pinder-web/src/Pinder.GameApi/Controllers/AdminPromptsController.cs:40` and `pinder-web/src/Pinder.GameApi/Controllers/AdminPromptsController.cs:64`, but the public `/api/*` surface is routed to FastAPI by `pinder-web/Caddyfile:47`, and the FastAPI router list exposes `/api/admin/prompts`, `/api/admin/prompts/save`, and `/api/admin/prompts/compile`, not `/api/admin/prompts/reload`. FastAPI only has the internal helper `_trigger_prompt_reload_async()` calling `GAME_API_URL/api/admin/prompts/reload` at `pinder-web/src/pinder-backend/routes/endpoints/admin_content_write.py:209`; there is no `@router.post("/prompts/reload")`.
**Impact**: Saving the Narrative Testbed prompt can persist the YAML via `putAdminContent`, then throw `Prompt saved, but reload failed (status 404)` from the extra frontend call. Admins get a false failure state and may believe the prompt did not reload, while the prompt PUT path already owns reload status reporting.
**Urgency**: U2 - topic default; this is an active React-to-FastAPI route gap on an admin workflow.
**Fixer-Agent Action Plan**: Remove the direct `fetch('/api/admin/prompts/reload')` from `NarrativeTestbed.handleSavePrompt` and surface the `reload_status` / `reload_error` returned by `putAdminContent`, matching `DramaticArcPromptEditor` and `AdminPage.promptsEditor`. Add a frontend test that saving the narrative prompt calls only `PUT /api/admin/content/prompts-narrative` and shows the warning path when `reload_status === 'failed'`. If a standalone reload button is still desired, add an authenticated/admin-gated FastAPI `POST /api/admin/prompts/reload` route that delegates to `_trigger_prompt_reload_async()` and cover it with a backend route test.

### Finding 2: GameApi Media Upload and Fetch Endpoints Are Hidden Behind the FastAPI /api Proxy
**File**: `pinder-web/src/Pinder.GameApi/Controllers/MediaController.cs:19`
**Issue**: `MediaController` defines `[Route("api/v1/media")]`, `[HttpPost("upload")]`, and `[HttpGet("{id}")]`, and the upload response emits `url = publicUrl` where `publicUrl` is built as `/api/v1/media/{assetId}` (`pinder-web/src/Pinder.GameApi/Controllers/MediaController.cs:99`). In production routing, however, `pinder-web/Caddyfile:47` sends every `/api/*` request to FastAPI on `127.0.0.1:5000`; `pinder-web/src/pinder-backend/main.py:477` imports only `admin, auth, characters, ops, sessions, visual_assets`, and `main.py:480-484` includes those routers. There is no FastAPI `/api/v1/media` proxy route and no React API client for `media/upload`, so the GameApi endpoint and the URL it returns are not reachable through the public web boundary.
**Impact**: Any photobooth/user-media feature calling `/api/v1/media/upload` from the SPA will hit FastAPI 404 instead of GameApi, and even a successful direct GameApi upload would hand back a public URL that Caddy routes to the wrong service. This makes the feature effectively unusable from pinder-web.
**Urgency**: U2 - topic default; backend functionality exists but is not wired through the deployed frontend/proxy boundary.
**Fixer-Agent Action Plan**: Choose one public boundary and make it explicit: either add FastAPI proxy routes for `POST /api/v1/media/upload` and `GET /api/v1/media/{id}` with auth, content-type, and size validation, or add a more-specific Caddy route before `handle /api/*` that proxies `/api/v1/media/*` to GameApi. Add integration tests that the public upload path reaches GameApi and that the returned `url` is fetchable through the deployed route. Then add or update the React media client/photobooth caller, or remove/feature-flag `MediaController` if the feature is not intended to ship.
