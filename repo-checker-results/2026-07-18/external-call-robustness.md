### Finding 1: Core non-streaming LLM clients use implicit HTTP budgets and unbounded provider bodies
**File**: `pinder-core/src/Pinder.LlmAdapters/Anthropic/AnthropicClient.cs:38`
**Issue**: The Anthropic path constructs `_httpClient = new HttpClient();`, then sends with `response = await _httpClient.PostAsync(ApiUrl, content, ct)` at `pinder-core/src/Pinder.LlmAdapters/Anthropic/AnthropicClient.cs:105` and reads the whole provider body with `var responseBody = await response.Content.ReadAsStringAsync()` at line 128. The OpenAI-compatible path has the same pattern: `_httpClient = new HttpClient();` at `pinder-core/src/Pinder.LlmAdapters/OpenAi/OpenAiClient.cs:33`, `PostAsync(url, content, ct)` at line 112, and `var body = await response.Content.ReadAsStringAsync()` at line 135. There is retry logic for 429/5xx, but no explicit per-call timeout policy, no response-size cap, and no bounded error-body read.
**Impact**: A slow or malicious LLM provider can hold gameplay/setup calls until implicit platform defaults or caller cancellation, and an oversized success/error body can be fully buffered into memory before parsing. Retries multiply that exposure during provider incidents.
**Urgency**: U1 - topic default; LLM calls are production turn/setup hot paths and can hang request/session progress or exhaust memory.
**Fixer-Agent Action Plan**: Introduce shared `LlmHttpOptions` with explicit request timeout, max response bytes, and max diagnostic bytes. Apply it when constructing internal `HttpClient`s and validate externally supplied clients or wrap sends in linked `CancellationTokenSource.CancelAfter`. Replace `ReadAsStringAsync` with a capped stream reader and add tests using fake handlers for oversized success bodies, oversized error bodies, timeout cancellation, and retry exhaustion preserving caps.

### Finding 3: Streaming LLM transports can wait forever after response headers
**File**: `pinder-core/src/Pinder.LlmAdapters/OpenAi/OpenAiStreamingTransport.cs:71`
**Issue**: The OpenAI streaming transport explicitly disables `HttpClient` timeout with `Timeout = Timeout.InfiniteTimeSpan` at lines 71-75, opens the SSE request with `SendAsync(..., HttpCompletionOption.ResponseHeadersRead, cancellationToken)` at line 145, and then waits on `reader.ReadLineAsync()` at line 242. Cancellation is observed, but there is no explicit first-byte/header budget, idle read timeout, maximum stream duration, or maximum frame size.
**Impact**: If a provider accepts the request and then stalls the SSE body without closing, the stream can hold a gameplay/setup pipeline and an HTTP connection indefinitely unless the caller happens to cancel. A large SSE frame can also grow the in-memory `StringBuilder` before parsing.
**Urgency**: U1 - topic default; streaming LLM setup/turn flows are user-visible and can wedge long-lived requests.
**Fixer-Agent Action Plan**: Add streaming-specific budgets: header timeout, idle line timeout, optional maximum stream duration, maximum SSE data-frame bytes, and provider heartbeat handling. ### Finding 4: Remote asset reads ignore configured payload caps when buffering upstream bodies

**Impact**: A compromised or misconfigured Eigencore backend can return a body far above the configured asset caps and force the caller to allocate it before validation or parsing. The comment-level cap does not provide a per-stream enforcement point on these reads.

### Finding 8: EF Core Postgres registration has no explicit command timeout or transient retry policy
**File**: `pinder-web/src/Pinder.GameApi/Program.cs:462`
**Issue**: Production persistence is wired with `opts.UseNpgsql(config.DatabaseUrl);` only. Repository calls pass cancellation tokens into EF operations, but the DB provider configuration does not set an explicit command timeout, connection timeout normalization, pool timeout, or bounded `EnableRetryOnFailure` policy for retryable Postgres failures.
**Impact**: Database behavior depends on Npgsql defaults or connection-string drift. A slow query/lock can hold request work until defaults fire, and transient network/database failures fail immediately instead of retrying under a small, controlled policy.
**Urgency**: U2 - de-escalated from U1 because EF/Npgsql has finite defaults and most repository calls pass cancellation tokens, but production DB robustness is still policy-by-default rather than explicit.
**Fixer-Agent Action Plan**: Normalize the database connection string with explicit connect/pool timeouts, configure `UseNpgsql(config.DatabaseUrl, o => { o.CommandTimeout(...); o.EnableRetryOnFailure(maxRetryCount, maxRetryDelay, ...); })`, and add integration tests/fake execution-strategy tests for command timeout, cancellation, and retryable vs non-retryable failures.

**File**: `pinder-web/frontend/src/api/core.ts:48`

**Urgency**: U2 - de-escalated from U1 because the blast radius is browser-side UX rather than server resource exhaustion.
**Fixer-Agent Action Plan**: Add a central `fetchWithTimeout` helper that composes caller signals with an `AbortController`, choose endpoint-specific budgets (short admin reads, longer LLM turn submits), cap text/JSON error reads where possible, and update `apiFetch`, `adminJsonFetch`, `deleteSession`, `shareClient`, and prompt admin APIs. Add tests for timeout aborts, caller abort precedence, 401 navigation behavior, and non-JSON error truncation.
