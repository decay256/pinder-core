### Finding 1: Existing-character smart randomize is wired in React and GameApi but missing from the FastAPI proxy
**File**: `pinder-web/frontend/src/api/adminClient.ts:10`
**Issue**: `generateRandomCharacter(existingCharacter)` switches from `POST /api/characters/generate-random` to `POST /api/characters/{id}/randomize` when the caller supplies an existing character at lines 10-13, and `CreationBench` passes the current edit key into that helper at `pinder-web/frontend/src/pages/CreationBench.tsx:581`. The downstream GameApi supports the capability with `POST /characters/{idOrSlug}/randomize` at `pinder-web/src/Pinder.GameApi/Controllers/CharactersController.cs:530`, but the public FastAPI character proxy only exposes generate-random, refresh-prompts, GET/PUT/DELETE, items, and anatomy routes in `pinder-web/src/pinder-backend/routes/characters.py:172`, `:230`, `:344`, and `:409`; there is no `/api/characters/{slug}/randomize` route to forward the SPA call.
**Impact**: The admin CreationBench "Smart Randomize" workflow for an existing saved character calls a route FastAPI does not implement, so the browser gets a 404 before the supported GameApi randomization operation can run. Brand-new random generation works, but in-place smart randomization is effectively unreachable from the deployed SPA.
**Urgency**: U2 - topic default; this is a live frontend API call unsupported by the public backend proxy.
**Fixer-Agent Action Plan**: Add a FastAPI `POST /api/characters/{slug}/randomize` proxy that mirrors the GameApi `/characters/{idOrSlug}/randomize` route, requires admin privileges like the other mutation/generation helpers, follows the same timeout/error/redaction behavior as `generate_random_character`, and add backend proxy coverage plus the existing frontend `characterMutations`/CreationBench tests.

### Finding 2: Backend OpenAPI still advertises removed setup-status endpoints
**File**: `pinder-web/contracts/backend-api.yaml:440`
**Issue**: The backend contract still documents `GET /api/sessions/{id}/setup-status` and `GET /api/sessions/{id}/setup-status/stream` at lines 440-489 and keeps the `SetupStatus` schema at line 1145. That contract text says FastAPI proxies the routes from GameApi and that the frontend polls or streams them. The actual GameApi route table deliberately removed those endpoints, with `RemovedSetupStatusEndpointsRegistrationTests` asserting they must not be registered at `pinder-web/src/Pinder.GameApi.Tests/Controllers/RemovedSetupStatusEndpointsRegistrationTests.cs:10` and `:38`; FastAPI likewise has no setup-status routes under `pinder-web/src/pinder-backend/routes/sessions.py` or `routes/endpoints/session_turns.py`, and the React sheet path now reads stake/background data from character sheets instead of setup-status.
**Impact**: Any client generated from `contracts/backend-api.yaml`, or any frontend engineer following the documented fallback, will call endpoints the backend intentionally no longer serves. Setup-progress behavior is now represented through create-session operation tracking and sheet/state fetches, but the published backend contract misdirects consumers to dead routes.
**Urgency**: U2 - topic default; this is a public backend contract promising unsupported API endpoints.
**Fixer-Agent Action Plan**: Remove the setup-status paths and `SetupStatus` schema from `backend-api.yaml`, or explicitly reintroduce matching FastAPI/GameApi routes if the contract is still intended. Update contract/openapi tests to assert the removed endpoints stay absent and keep the React `SetupStatus` type only if a backward-compatibility fixture still needs it.
