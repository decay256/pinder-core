### Finding 1: Adapter phase calls bypass the central temperature catalog
**File**: `pinder-core/src/Pinder.LlmAdapters/PinderLlmAdapter.cs:677`
**Issue**: `GetSuccessImprovementAsync`, `GetSteeringQuestionAsync`, and `GetHorninessQuestionAsync` pass raw temperatures directly into `SendWithDiagnosticsAsync`: `0.8` at line 677 and `0.9` at lines 729 and 768. The same class already centralizes other phase choices through `_options.DialogueOptionsTemperature ?? LlmPhaseTemperatures.DialogueOptions`, `_options.DateeResponseTemperature ?? LlmPhaseTemperatures.DateeResponse`, and `_options.DeliveryTemperature ?? LlmPhaseTemperatures.OverlayRewrite`, so these three production calls sit outside the existing config path.
**Impact**: Changing the host-level LLM temperature or the `LlmPhaseTemperatures` defaults will not affect success-improvement, steering, or horniness-question generation. Those gameplay LLM phases can drift from the rest of the adapter without an obvious config knob or startup validation catching it.
**Urgency**: U2 - escalated from U3 because these are production gameplay LLM calls and config changes can leave them running with stale sampling behavior.
**Fixer-Agent Action Plan**: Add central phase settings for success improvement, steering, and horniness question generation, either as new `PinderLlmAdapterOptions` nullable overrides backed by `LlmPhaseTemperatures` constants or by reusing the existing generic/default option intentionally. Replace the three raw literals and add focused adapter tests proving non-default configured temperatures reach the transport for these methods.

### Finding 2: Admin speculation copies phase temperatures instead of using shared constants
**File**: `pinder-web/src/Pinder.GameApi/Services/SessionSimulationService.cs:200`
**Issue**: The speculative adapter wiring sets `DeliveryTemperature = 0.7`, `DialogueOptionsTemperature = 0.9`, and `DateeResponseTemperature = 0.85` inline even though pinder-core exposes `LlmPhaseTemperatures.OverlayRewrite`, `DialogueOptions`, and `DateeResponse` for these exact defaults. This lives beside config-driven `MaxTokens = _config.LlmMaxTokens` and `Temperature = _config.LlmTemperature`, so only the phase-specific values are copied.
**Impact**: Admin speculation can diverge from normal session behavior whenever the shared phase defaults change. Operators also cannot tune these per-phase speculation temperatures through the same central config surface used for max tokens and the generic temperature.
**Urgency**: U3 - topic default; this is configuration drift on an admin LLM path, but the copied values currently match the shared defaults.
**Fixer-Agent Action Plan**: Replace the literals with `LlmPhaseTemperatures` constants or add explicit `GameApiConfig` fields for speculation phase temperatures and populate them in `Program.cs`. Add a regression test around `SessionSimulationService` adapter creation so speculation follows the shared/default config values rather than local numbers.

### Finding 3: Character generation hides a hardcoded 2048-token floor from LLM config
**File**: `pinder-web/src/Pinder.GameApi/Services/CharacterGenerator.cs:161`
**Issue**: Character generation wraps the configured token budget with `Math.Max(2048, _config.LlmMaxTokens)` when constructing `PinderLlmAdapterOptions`, then repeats the same floor in the actual transport call at line 192. The central setting is documented and wired as `GameApiConfig.LlmMaxTokens` / `LLM_MAX_TOKENS`, but this production LLM workflow has an unadvertised minimum embedded in the service.
**Impact**: An operator lowering `LLM_MAX_TOKENS` for cost, latency, or model compatibility will still send at least 2048 tokens for character generation. If character generation genuinely needs a larger budget, that policy is not visible in startup config, appsettings, or validation, so it can drift from the rest of GameApi's LLM budgeting.
**Urgency**: U2 - escalated from U3 because this is a production admin LLM call and the hidden token floor can change provider cost/latency despite explicit configuration.
**Fixer-Agent Action Plan**: Introduce a named config value such as `CharacterGenerationMaxTokens` or `CharacterGenerationMinTokens`, validate it in `Program.cs`, document the env/appsettings key, and use it in both the adapter options and `SendAsync` call. If no special budget is required, remove the `Math.Max(2048, ...)` wrapper and pass `_config.LlmMaxTokens` directly.

### Finding 4: Concrete transports duplicate optional temperature and token defaults
**File**: `pinder-core/src/Pinder.LlmAdapters/Anthropic/AnthropicTransport.cs:81`
**Issue**: Both changed concrete transports declare their own optional defaults: `AnthropicTransport.SendAsync(..., double temperature = 0.9, int maxTokens = 1024, ...)` and `OpenAiTransport.SendAsync(..., double temperature = 0.9, int maxTokens = 1024, ...)` at `pinder-core/src/Pinder.LlmAdapters/OpenAi/OpenAiTransport.cs:66`. These duplicate the `ILlmTransport` optional defaults and sit alongside `PinderLlmAdapterOptions.Temperature = LlmPhaseTemperatures.Default` / `MaxTokens = 1024`.
**Impact**: Call sites typed as a concrete transport can omit temperature or token arguments and silently use provider-local literals instead of the host's central options. The values match today, but future tuning has to update the interface, both transports, and the adapter options in lockstep.
**Urgency**: U3 - topic default; this is drift-prone API surface, but current production adapter calls pass explicit values.
**Fixer-Agent Action Plan**: Remove optional defaults from concrete transport implementations where possible and require callers to pass explicit values, or route all default values through one shared constant/config type. Add a compile-time or focused unit test that direct concrete transport calls used by tests/tools do not rely on provider-local optional defaults.
