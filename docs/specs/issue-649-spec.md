# Specification: Active Archetype Selection and Injection (Issue #649)

**Module**: `docs/modules/characters.md`

## Overview
This feature replaces continuous level ranges with discrete `ArchetypeTier` definitions to formalize character archetype progression. The `CharacterAssembler` now computes a character's tier from their level and selects the highest-frequency eligible archetype as the single `ActiveArchetype`. This active archetype, along with its YAML-sourced behavior instruction, is stored on the `CharacterProfile` and injected into the LLM system prompt via `PromptBuilder`. This allows the LLM (and `LlmPlayerAgent`) to drive dialogue option generation based on the active persona constraints.

## Input/Output Examples

**Tier Mapping**:
- Input: `characterLevel = 2` -> Output: `ArchetypeTier.Tier1`
- Input: `characterLevel = 5` -> Output: `ArchetypeTier.Tier2`
- Input: `characterLevel = 10` -> Output: `ArchetypeTier.Tier4`

**Active Archetype Selection**:
- **Setup**: Level 4 character (Player Tier = `Tier2`).
- **Equipped Archetypes** (from items): 
  - "The Hey Opener" (Count: 3, Tier 1)
  - "The Player" (Count: 2, Tier 4)
  - "The Oversharer" (Count: 1, Tier 2)
- **Evaluation**: "The Player" is Tier 4, which exceeds the player's Tier 2. It is filtered out. The remaining are "The Hey Opener" (Count 3) and "The Oversharer" (Count 1).
- **Result**: `ActiveArchetype` is "The Hey Opener" because it has the highest count among eligible archetypes (Tier <= Player Tier).

## Acceptance Criteria

1. **ArchetypeTier Enum**: Introduce `ArchetypeTier` (`Tier1` to `Tier4`) in `Pinder.Core.Characters`.
2. **ArchetypeDefinition Update**: Replace `MinLevel` and `MaxLevel` in `ArchetypeDefinition` with `ArchetypeTier Tier` and `string BehaviorInstruction`.
3. **ArchetypeCatalog Update**: Update hardcoded definitions in `ArchetypeCatalog` to match discrete tiers based on original level mappings:
   - Levels 1-3 = `Tier1`
   - Levels 4-6 = `Tier2`
   - Levels 7-9 = `Tier3`
   - Levels 10+ = `Tier4`
   Also add placeholder or known YAML behavior instructions for each archetype.
4. **Assembler Logic**: 
   - `CharacterAssembler` must calculate the `playerTier` from the `characterLevel` parameter using the 1-3/4-6/7-9/10+ rule.
   - It filters the ranked archetypes by selecting only those where `ArchetypeDefinition.Tier <= playerTier`.
   - It promotes the eligible archetype with the highest count to `ActiveArchetype` and includes it in `FragmentCollection` and `CharacterProfile`.
5. **Prompt Injection**: `PromptBuilder.BuildSystemPrompt` must inject the `ActiveArchetype.BehaviorInstruction` (if an active archetype is set) into the LLM system prompt text (e.g., in the "ARCHETYPES" block or a new "BEHAVIOR" block).
6. **SessionRunner Context**: `PlayerAgentContext` must expose `ActiveArchetype`, `TextingStyleFragment`, and `History` properties so the `LlmPlayerAgent` can use them to generate character-aware choices.
7. **Playtest Output**: The session runner's `Program.cs` console output must be updated to display the `ActiveArchetype` and any `ShadowGrowthWarning` generated by the `ScoringPlayerAgent`.
8. **Scoring Adjustments**: `ScoringPlayerAgent` mathematically weights failure cost vs success gain independently.

## Edge Cases

- **Ties in Archetype Count**: If two eligible archetypes have the same count, resolve by picking the one with the higher Tier. If Tiers are also tied, resolve alphabetically by name.
- **Level <= 0**: If character level is 0 or negative, default the calculated tier to `Tier1`.
- **Unknown Archetypes**: If a fragment provides an archetype name not found in `ArchetypeCatalog`, it cannot be evaluated for Tier and should be ignored when selecting the `ActiveArchetype`.
- **No Eligible Archetypes**: If a character has no items, or all equipped archetypes exceed their Tier, `ActiveArchetype` is set to `null`.
- **Level > 10**: Any level 10 or greater maps to `Tier4`.

## Error Conditions

- Passing a `null` string to `ArchetypeCatalog.GetByName` returns `null` safely.
- In `PromptBuilder`, if `ActiveArchetype` is `null`, the builder must not throw an exception and should omit the behavior instruction block gracefully.
