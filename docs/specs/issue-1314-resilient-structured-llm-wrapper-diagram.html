Pinder Architecture & Control Flow

Speculative Fast-Path vs Synchronous Slow-Path & Proposed Resilience Wrap (Issue #1314)

1. Why the Slow Path Wedges

  • • **Synchronous LLM Block:** On the slow path, turn resolution runs in real time, locking the session's stream-slot for 10-30s during the API call.
  • • **Reconnection Rejections (HTTP 429):** Any transient client network drop triggers a reconnect that gets rejected with a 429 because the background lock is still held.
  • • **Hardcoded Rejection (HTTP 400):** The frontend hard-maps all stream errors to 400 Bad Requests, aborts immediately, and drops the game options panel, wedging the player interface.

2. Why Fast-Gameplay Succeeds

  • • **Speculative Pre-generation:** While the player is reading, the backend uses parallel execution branches to pre-generate the next turn's outcomes.
  • • **Instant Adoption (<100ms):** When the player clicks an option, the pre-generated outcome is instantly adopted, removing any active LLM blocking times.
  • • **Connection Immunity:** Because the resolve resolves instantly, there is virtually zero time window for network timeouts or browser connection drops.

3. The New Resilience Wrapper (Issue #1314)

  • • **Stateless Interception:** Wraps `PinderLlmAdapter.cs` calls in a 3x retry loop. Intermediate formatting failures (e.g. chain-of-thought bleed) are captured safely in memory.
  • • **Unpolluted History:** If attempts 1 or 2 fail, the invalid output is discarded. Only successful, fully-validated outputs are written to the database history or session state.
  • • **Exponential Backoff:** Configurable `MaxContractViolationRetries` and `ContractViolationBackoffMs` parameters ensure graceful delays between attempts, keeping active play unblocked.
